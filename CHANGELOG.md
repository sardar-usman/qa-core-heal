# Changelog

## 0.2.0

**Run-first healing is the new default.** `qa-core-heal <spec>` now runs the spec with Playwright before touching anything. All tests green prints "All tests passing. Nothing to heal." and exits 0 — no scanning, no probing, no browser launched. On failures, each one is classified from the JSON report: locator failures (a timeout waiting for a locator, an element not found, a strict mode violation) are heal candidates; everything else — an assertion value mismatch on an element that WAS found, a navigation or network error, a thrown app error — is reported as "not a locator problem, healing won't fix this" with the failure line, and never healed. For each locator failure the CLI extracts the failing selector from the error message and the page URL the test was on when it failed from the Playwright trace (tests run with `--trace retain-on-failure`), then probes ONLY those selectors on those pages — the same heal pipeline and refusal rules as always. After the usual diff-and-confirm, only the previously failing tests are re-run to verify, and applied heals land in the audit log with the verified flag.

Because run-first probes the page a test was actually on when it failed, it can heal locators the static probe honestly cannot: an element on a page only reached by clicking through (no `goto()` ever names it) refuses under the static probe and heals under run-first — locked by a new eval scenario, alongside passing-suite-means-zero-probing, locator-failure-heals-and-re-runs-green, and assertion-failure-is-never-healed.

**The static probe moved behind `--scan`.** No test execution, every locator in the spec and its page objects probed on its inferred route, current output — for locator audits and suites too expensive to run.

**Prompt input fix.** The apply prompt accepts y/Y/yes, declines on n/no/empty, and re-prompts on anything unrecognized instead of silently treating it as "no". The `--yes`/`-y` flags and the exit-code contract are unchanged (0 success, 1 error, 2 heals available but not applied; "all tests passing" is a 0).

**Failing selectors are matched to source structurally, never textually.** Playwright renders a failing locator its own way (`getByRole('textbox', { name: 'Emails' })`) while the source may say `page.getByRole("textbox", {"name":"Emails"})` — quote style, spacing, quoted-vs-bare option keys, receiver prefix all differ. Both sides now parse into a canonical structure (method + role/selector + options) and compare structurally, so formatting can never make a detected failure silently match nothing. The failure's stack trace is the primary targeting signal: the structurally-matching locator call nearest the error's source location wins (Playwright errors carry the file:line of the action call), with structural matching across the whole import graph as the fallback — which is how a failure raised in a spec heals the locator defined in an imported page object. When neither signal finds the call, the CLI never claims "Nothing to heal" while tests are red: it prints `1 failing locator could not be matched to source: <selector> (from <test title>). This is a bug worth reporting.` and exits 1 (the `--json` payload lists these under `unmatchedFailures`). A run whose Playwright report contains zero tests is likewise an error, not a clean bill of health. New eval scenarios pin the three matching shapes end-to-end: a broken locator living in an imported page object, a getByRole with an options object across the quote/spacing mismatch, and a plain CSS locator().

**Config loading matches Playwright's ESM/CJS interop.** Real-world configs mix ESM imports with `require("dotenv").config()`; Playwright's loader supports that, but the child-process native import parsed the file as ESM where `require` is undefined, threw ReferenceError — and the loader swallowed it, wrongly reporting "no baseURL". The child loader now exposes a CJS-compatible `require` (created against the config's own path, so it resolves the user's node_modules) before importing, loads genuinely-CJS configs (`module.exports`, no import/export syntax) via `require()` directly instead of `import()`, and preloads `.env` from the config's directory without overriding already-set env vars (configs that load dotenv themselves now simply work). The loader's result travels on a dedicated pipe (fd 3), so a config — or dotenv's own logging — writing to stdout can never corrupt it. Diagnostics are now mandatory: a config that throws or times out produces a warning with the underlying error ("playwright.config.ts failed to load (ReferenceError: require is not defined); falling back to goto() scan / --base-url") — absence is never reported when the truth is failure. Project-level baseURLs: when a config's projects agree on one baseURL it is used; when they disagree, the CLI warns and the new `--project <name>` flag chooses. Regression fixtures pin the exact ESM+require+dotenv+conditional-reporters pattern, a pure-CJS config, the throwing-config warning text, project disagreement, and env-var precedence.

**Classification is evidence-based, not message-shape-based.** Real case: a locator wait outlived the TEST timeout, teardown closed the browser, and the pending action died with "Target page, context or browser has been closed" — the top-level error said only "Test timeout of 30000ms exceeded" and the run was misclassified as non-locator. All errors of a failed result are now collected (on a test timeout the pending locator action arrives as a secondary error) and a failure is a locator failure whenever the evidence exists ANYWHERE in it: a `locator.<action>:` / `expect.<matcher>:` prefix or a call-log "waiting for locator/getBy..." line, regardless of the top-level message. The selector comes from the call log; source locations come from every error's location and project stack frames (node_modules skipped). The one earlier gate stands: a locator that RESOLVED to a real element stays non-locator. A timeout with no locator evidence anywhere reports "timed out with no pending locator action".

**Failure-URL fallback chain for custom browser setups.** Manual `chromium.launch()` in beforeAll can produce no usable trace, leaving the failure URL unknown. The chain is now: trace, else the statically inferred route for that locator (its file's goto/override/inherited routes — the bare base-URL fallback no longer counts as a route signal), else an explicitly given `--base-url`, else a loud refusal: "could not determine the page where \<selector\> failed (custom browser setup produces no trace); re-run with --route \<file\>=\<route\> or --base-url". The wrong page is never probed silently. A new `--no-trace` flag skips tracing entirely for setups where it is unavailable or unwanted. Page-object imports are now followed recursively, so a locator defined in a widget two imports below the spec is matched and healed. Three new run-mode eval scenarios pin all of this: the timeout+closed shape classified and healed, a manual-browser spec healing via route inference, and a two-imports-deep POM locator healing end-to-end.

**Trace URLs come from the MAIN frame, with sanity checks.** On a Vercel-hosted app the extracted failure page was the vercel.live feedback widget — a child iframe whose snapshot happened to be the last one in the trace. Extraction now uses the trace's isMainFrame flag (last main-frame snapshot wins; a missing flag from older trace formats counts as main) and skips non-http URLs like about:blank and chrome-error://. On top of that, a trace URL whose origin matches neither the resolved base URL nor any goto()/--route target is discarded with a stderr message saying why, falling back to route inference — the wrong page is never probed. The run-mode fixture now embeds a different-origin iframe (with a mutation loop that deliberately makes its snapshot the LAST one recorded) and asserts the extracted URL is the main page.

**Config loading survives CommonJS package contexts.** The same real-repo config that previously hit "require is not defined" then failed with "SyntaxError: Cannot use import statement outside a module": in a package whose package.json has no "type" field (or says "commonjs"), some Nodes execute a stripped .ts config as CJS, where its import statements are illegal. The loader now always tries native import() first (require shim installed); when that fails for a .ts config it retries with an ESM-forcing load hook (module.register + stripTypeScriptTypes) that serves the file as a stripped ES module regardless of package type; the createRequire fallback is reserved for files with no import/export syntax; and when every attempt fails the diagnostic reports each attempt's error. Two more real-world fixes fell out of reproducing this faithfully: the earlier fixture had no package.json at all (inheriting "type": "module" from this repo — why it passed while reality failed), and dotenv's tip line — which itself contains a "{" — printed ahead of Playwright's JSON report, breaking the naive report slicing; report parsing now scans for the actual report object. Regression fixtures: the byte-for-byte config in a typeless package AND under explicit "type": "commonjs" (deterministic on every Node), both-errors diagnostics, and an end-to-end run-mode heal through the exact CLI code path with a relatively-navigating spec, where healing is impossible unless the config's baseURL truly resolves.

**Type-only named imports survive the ESM retry.** A config importing a TypeScript type as a plain named import ("import { defineConfig, devices, ReporterDescription } from '@playwright/test'") crashed the ESM retry with "does not provide an export named 'ReporterDescription'": type stripping removes annotations but does NOT elide type-only import specifiers written without the `import type` keyword. The retry hook now rewrites every named import (including default+named mixes) to a namespace import plus destructuring — a missing export destructures to undefined, which is harmless because a type identifier is unused after stripping. Regression fixture: that exact config shape in a "type": "commonjs" package, resolved at the loader level and healed end-to-end through the run-mode CLI path.

**The trace URL is trusted when nothing else exists.** The origin sanity check discarded a correct main-frame failure URL precisely when it was most needed: the config had failed to load, no baseURL resolved, so there was nothing to compare origins against — and the run died with "could not determine the target URL" while holding the right answer. The origin check now only applies when an anchor exists (a resolved baseURL or an absolute goto()/--route target); with none, the main-frame trace URL — which already passed the non-http and child-iframe filters, which always apply — is trusted, the base URL is derived from its origin, and the output notes "using failure page from trace: \<url\>". A new run-mode eval scenario locks it: a config Playwright loads but heal cannot (a TS enum defeats type stripping), a spec that only navigates relatively, and a heal that succeeds purely from the trace URL — alongside the load-failure warning.

**Authenticated probing via --storage-state.** On an auth-gated app, run mode classified the failure and extracted the right failure URL, but the unauthenticated probe got redirected to the login screen and refused against the wrong page — everything worked except auth. `--storage-state <path>` (and the existing qa-core.config.json `auth.storageState`) now applies Playwright's storageState to the probing context in both run mode and `--scan`. When neither is set, heal auto-detects in priority order — `use.storageState` from the playwright config (top-level or agreed across projects), then the conventional `.auth/state.json` and `playwright/.auth/user.json` — announcing "using storage state from \<path\>"; detection finding nothing just probes unauthenticated as before. Probes are now redirect-aware: a page that lands somewhere other than the requested URL is NEVER probed as if it were the target. The output says "requested /accounts, landed on /login (redirected)", adds "the page may require authentication; pass --storage-state \<path\>" when no session was used and the landing page looks like a login route, or "storage state was applied but /accounts still redirected to /login; the saved session may be expired. Re-generate it and retry." when one was — and the locator refuses with the redirect named. Storage-state file contents never appear in heal output or the audit log; only the path is logged. Ambiguity refusals now NAME their candidates (capped at 5): "several elements match the intent (candidates: #alpha, #beta, #gamma), refusing to guess". A three-part eval scenario locks the feature against a fixture server that 302-redirects /app to /login without a session cookie: unauthenticated refusal with the redirect and hint, an end-to-end heal with a valid saved session, and the expired-session message with a stale one.

**--auth-setup: authenticated probing via the user's own login code.** Complementing --storage-state: `--auth-setup <file>#<export>` (default export when #export is omitted; also `authSetup` in qa-core.config.json) points at the project's existing login helper. Heal loads the module with the same TypeScript machinery as the config loader — require shim, ESM-forcing retry, in-process because the function needs the live page — and calls `await fn(page)` on a fresh page in the probing context before probing, in both run mode and --scan. It deliberately does NOT parse or re-execute beforeAll/beforeEach hooks: setup code can seed data or trigger side effects, and only the user can say what is safe to re-run. A login that throws or hangs (60s default, `--auth-setup-timeout <seconds>`) fails the run loudly with "auth setup \<file\>#\<export\> failed: \<error\>" — probing unauthenticated is never a silent fallback. `--auth-setup` takes precedence over `--storage-state` (with a note), the single probing context carries the session across every probed route (one login, many probes), and redirect detection stays live with its own diagnosis: "auth setup ran but /accounts still redirected to /login; the login function may have failed silently or the session did not persist". Only file paths, export names, and pass/fail are logged; credentials, cookie values, and storage contents never are. The auth eval fixture gained a POST /api/login endpoint and a three-part scenario: a real login function heals end-to-end, a throwing one fails loudly with nothing probed, and a silently-failing one gets the redirect-after-auth message. The README's Authenticated apps section now carries the decision guide: --auth-setup pointing at your login helper (recommended), --storage-state for saved sessions and CI, and the save-auth script pattern.

**Confirmation is churn-tolerant and its refusals are debuggable.** A correct heal ([name="search_field_1"] → the page's [name="search_field"]) was refused with a bare "re-resolved to a different element". Reproducing on an interval-re-rendering page failed 10/10 — deterministically, which pointed past the node-churn hypothesis to the real root cause: the confirmation matcher compared the spaced token ("search field 1") against the raw attribute ("search_field") and separators never matched. Matching now compares separator-normalized forms too (underscores/dashes/spaces are the same identity word sequence). Churn tolerance was layered in as well, since the repro also proved nodes get replaced mid-probe: a failed confirmation attempt retries once after a 250ms settle, and then a fingerprint pass takes over — two FRESH locator reads compared as logical identity (tag, key attributes, accessible-name sources, geometry within 5px) rather than node identity, so a React re-render replacing every node cannot fail a correct heal. Genuine mismatches still refuse, and the refusal finally names what differed: "re-resolved element differs: expected an element matching \"search panel 42\" (from page.locator('#search-panel-42')), got input\[id="global-lookup"\]\[type="search"\] at (x, y); not healing". The pinned-handle mutation-watch protections are untouched — the hostile-mutation suite still gets its deterministic instability refusal. A new react-churn eval suite locks both sides: the heal surviving 150ms-interval node replacement end-to-end, and a genuinely different control refused with the diff named. (Fixture note: the suite lives on port 4191 because 4190 is on the fetch-spec bad-ports list — Chromium and undici both refuse it.)

**The probing context carries the resolved base URL.** Login helpers written for Playwright test contexts navigate relatively (`page.goto("/")`) and rely on use.baseURL; the probing context did not set it, so --auth-setup died with "Cannot navigate to invalid URL". The context is now created with `baseURL` set to whatever resolution produced (config / trace-derived / --base-url) — always, not just for auth-setup, so relative navigation anywhere in probing resolves like it does in the user's tests. If a future path ever probes without a resolvable base and a login helper then fails on relative navigation, the failure says: "your login function navigates to a relative URL but no baseURL could be resolved; pass --base-url". The auth-setup fixtures now navigate relatively on purpose, pinning this.

**In-process module loading is warning-free.** The child-process config loader suppresses Node's TypeScript noise with --no-warnings, but the in-process auth-setup loader leaked ExperimentalWarning (Type Stripping / stripTypeScriptTypes) and module-type warnings into CLI output. A process-warning filter now drops exactly those warning shapes for the duration of the load and restores the original emitter after; the auth-setup success test asserts the real stderr stream is clean of them.

**Token containment, glued digit suffixes, and render-random ids.** A deterministic refusal on a React SPA: getByRole("textbox", {name:"search1"}) refused every run with a near-miss naming "_r_NN_-form-item" — a different render-random id each run. Verified on a modeled fixture: the field's FULL accessible name is "Search by account or login" (Playwright name matching is substring, which is why a prior heal to name:"search" resolved green), whole-string fuzzy scoring of "search1" against it lands far below the bar, and the glued digit could not strip ("search1" is one word). Four fixes: (1) suffix stripping now handles digits glued to the word — search1 → search, email2 → email — so the ladder retries the stripped name first; (2) token-containment scoring — a stripped identifier matching a whole word of a candidate's accessible name scores above the fuzzy band (below exact); for id/name/testid values the identifier must also cover at least half the value's meaningful words, so "result" inside "newsletter-result-block" is NOT identity (the wrong-heal trap of widening id matching, pinned by a unit test); two candidates containing the token refuse as ambiguous with both named; (3) render-random identifiers (React useId patterns like _r_17_-form-item / :r1a:, and all-hash values) contribute nothing to matching and are excluded from refusal naming; near-misses now name the top 3 candidates with scores ("closest candidates below the confidence threshold: #emailz9 (0.71)"), falling back to a generated id only when nothing else exists; (4) hardening: a mutation-quiet settle (150ms quiet, capped at 2s, `--settle-ms` to change, 0 disables) runs before candidate collection, noting "candidate collection may be incomplete" when the cap is hit. Same-element confirmation and the element fingerprint now read the ASSOCIATED LABEL text (el.labels) — for label-for inputs that is the accessible name, and often the only human identity the element carries. New eval cases: the long-accessible-name heal and a two-field token-ambiguity refusal, both on Mantine-style markup with useId ids. pricing-hostile was run specifically to confirm token containment re-opens none of the hostile intent-widening cases.

In `--json` run-first output, the payload gains a `nonLocatorFailures` array listing the failures healing cannot fix.

## 0.1.5

Two gaps found in real testing, plus CLI ergonomics.

**CI-friendly apply.** `--yes` gained a `-y` alias. When stdout is not a TTY (or with `--json`) the CLI never prompts: with `--yes` it applies; without it the run becomes preview-only — the full proposed diff still prints — and exits with code **2**, meaning "heals available but not applied", so pipelines can detect actionable heals without ever writing files by accident. Declining the interactive prompt exits 2 as well. Exit codes are now documented: 0 success (applied, nothing to heal, or `--dry-run`), 1 error, 2 heals available but not applied. In `--json` output, `dryRun` is now true whenever the run wrote no files.

**Readable multi-file output.** Per-locator lines (broken / healed / unhealable with reason) are grouped under the source file they live in — one block per spec or page object instead of a flat stream — with the totals kept at the end.

**Fuzzy and suffix-strip now cover semantic identities, not just CSS.** The typo pipeline used to apply only to simple `#id` / `.class` / test-id selectors, so `page.getByRole("textbox", { name: "Ema_il_2" })` refused even though normalized "email2" is 0.833 similar to "Email" — above the 0.8 bar. The string identity of semantic locators — the getByRole name, getByLabel, getByPlaceholder, getByText, getByTitle, getByAltText strings, and the `{ hasText }` filter on locator() calls — now runs through the same pipeline in the same order: the identity as written, a humanized form (separators and camelCase to spaces), the auto-suffix-stripped form, then fuzzy; every candidate still faces the kind guard, the ambiguity rules, and same-element confirmation. To let text identities match, the fuzzy page scan now also collects the visible text of leaf elements (label elements excluded — their text already counts as the identity of the control they label). The real case heals on the live shop: the mutated role name resolves to getByRole("textbox", { name: "Email" }) on /login, and a mutated role name whose normalized form sits equally close to two inputs refuses as ambiguous with both candidates named (both pinned in the eval suites). The hostile fixtures were measured before enabling this: their reworded texts score 0.21–0.66 against the retired strings, safely below the heal bar, and they stay refused.

**Never-found refusals now say only what the engine actually knows.** The state-dependent phrasing used to fire as a catch-all whenever an element was missing and nothing similar existed, which produced a false explanation for plainly static elements. The wording is now a hierarchy: a below-threshold fuzzy near-miss keeps the "closest candidate X below confidence threshold" reason; a selector token that names an after-action UI pattern (toast, modal, alert, result, notification, snackbar, dialog, popup, confirmation, flash) is positive evidence and refuses with "element may be state-dependent (selector token \"result\" suggests it appears only after user actions); static healing cannot verify it"; with neither, the refusal hedges honestly: "not found on route /login: no matching or similar element on the probed page. The element may have been removed, renamed beyond recognition, or may only appear after user actions." The pinned .result eval reason moved to the evidence-carrying wording.

## 0.1.4

Everything below shipped together in 0.1.4 (an internal 0.1.3 was never published). The work came from healing a real multi-page POM project against `https://demowebshop.tricentis.com`, then testing the result in anger.

**Base URL resolution.** The CLI demanded --base-url even when the target project already declared its base URL. It now resolves the URL in priority order: an explicit --base-url / qa-core.config.json baseUrl, then use.baseURL from the project's own playwright.config.ts/.js — actually imported (defineConfig runs and env vars evaluate; no text scraping), then the first absolute page.goto() in the specs. Only when all of those fail does it ask for --base-url. Relative goto() routes are joined with the resolved base URL. The config import runs in a child Node process launched with --no-warnings, so the noise Node prints when loading TypeScript (ExperimentalWarning: Type Stripping) or a typeless package.json (MODULE_TYPELESS_PACKAGE_JSON) never reaches the CLI output, and a config that throws cannot take the run down. TypeScript configs need Node's native type stripping: the package now declares engines.node >= 22.6, and on older Node the CLI exits with "qa-core-heal requires Node 22.6+ to read TypeScript configs. Detected \<version\>. Workaround: pass --base-url and --route flags." — but only when the .ts config is actually the thing standing between it and a base URL; .js configs and explicit flags keep working.

**Per-locator page context.** Every locator used to be probed against a single page (the base URL), so locators living on other routes came back "could not be re-resolved" or "ambiguous" — and could even heal to a same-named element on the wrong page. Each locator is now probed on the route its spec navigates to: routes are inferred from page.goto() calls (nearest goto above the locator, else the file's first), page objects with no goto inherit the routes of every spec importing them, and a page object shared by specs with different routes is probed on each route and healed only when all routes agree on the same replacement. Specs are healed jointly in one pass, so shared page objects are scanned once instead of once per importing spec. A new repeatable `--route <file>=<route>` flag overrides the inference per file. Refusals now say where the engine looked: "not found on route /login: could not be re-resolved there" is distinct from "ambiguous on route /login: several elements match the intent, refusing to guess" and from "candidates disagree across routes (...); refusing to guess".

**Element-kind guard.** Real case: #register-button — a submit button on /register — was healed to getByRole("link", { name: "register" }), the nav link in the header. The engine now infers the expected element kind from the original selector (whole-word tokens like button, link, input, checkbox in ids/classes, the getByRole role, and the API used on the locator: .fill implies a text input, .check a checkable, .selectOption a select) and reads the candidate's actual kind from the live element. A definite conflict refuses with "kind mismatch: expected button, candidate is link" instead of healing. A refusal here is correct behavior; the engine never guesses across kinds.

**Typo'd identifiers heal through a fuzzy stage.** `#Emai_l` (a typo of `#Email`) used to refuse: its tokens never match "Email" semantically, and the auto-suffix retry (which handles renamed generated ids like "#Email_1" by stripping the trailing fragment) does not cover in-word typos. A new fuzzy stage — running strictly AFTER exact and suffix-stripped intent matching fail, so it always scores lower — normalizes the broken identifier to bare lowercase alphanumerics and scores it against the page's ids, name attributes, and accessible-name sources (aria-label, associated label text, placeholder, test ids) by edit-distance ratio with adjacent transpositions counted as one edit. A single candidate at or above 0.8 similarity heals, still subject to the element-kind guard, the ambiguity rules, and same-element confirmation; two candidates in the band refuse as "ambiguous on route X: several close matches (#a, #b), refusing to guess"; a candidate below the bar is named honestly: "not found on route X: closest candidate \"#Email\" below confidence threshold". Fuzzy only considers simple `#id` / `.class` / test-id selectors — a compound selector names a position, not an identity.

**State-dependent elements are documented, not forced.** `.result` on /register exists only after the form is submitted; static probing can never see it and its refusal is CORRECT. Such never-found locators now refuse with a distinct reason — "element may be state-dependent (appears only after user actions); static healing cannot verify it" — and the README gained a Limitations section explaining this honestly, with runtime healing (probing at the moment a selector fails mid-test) as the planned answer.

**Refusal reasons are printed in every mode.** The default (apply) mode ran its preview probe silently, so a run that ended in refusals printed only "Nothing to heal ... N refused" with no detail, while --dry-run showed full reasons. The preview pass now prints the same per-locator lines as dry-run — intact, healed, and unhealable with the reason — in all human modes; the confirmed write pass no longer re-prints them. JSON output is unchanged.

Eval coverage grew from 6 to 8 suites (70 locators, 34 deliberately broken), including demowebshop-pom, which runs against the live Tricentis demo shop: base URL only in playwright.config.ts, #Email_1 and #Emai_l both healing to getByRole("textbox", { name: "Email" }) on /login, #register-button pinned as intact and never healed to a link, and .result pinned as refused with the exact state-dependent reason. The new fuzzy-typos suite pins the fuzzy kind-conflict refusal and the close-match ambiguity refusal. A node:test unit/integration suite (npm test) covers the config loader, the Node version gate, route inference, fuzzy scoring, the kind guard, and CLI output. Final scoreboard: 25/25 heals verified by passing re-runs, 9/9 refusals with the expected reasons, 1/1 intact locator untouched, 0 wrong heals.

## 0.1.2

Fixed a nondeterministic heal verdict on pages that rewrite identity attributes on a timer. On a page where two buttons swapped their id and title attributes every millisecond, the engine found the right element on every run, but the final confirmation read the element's attributes at a single instant, so the verdict depended on which phase of the swap that read landed in: some runs healed, others refused, from the same command on the same page.

The confirmation now pins the element once, reads its identity evidence twice over a short window, and watches for mutations in between. Any evidence that changed during the window is discarded before the match runs, including values the mutation watch saw pass through between the two reads. Stable evidence confirms exactly as before, so all previous eval results are unchanged. When the only evidence that would have matched was unstable, heal now refuses deterministically with the reason "element identity attributes are unstable; cannot confirm the match" instead of guessing.

A new eval fixture, hostile-mutation, locks the behavior: ten consecutive dry runs produce byte-identical output. The bug was found via a community challenge page.
